## 一、Google论文：4万个Skills只有2.5%能用

Google 开发者和研究团队今年 2 月发了一篇论文《SkillsBench》，对当前 Agent Skills 生态做了一次系统性的摸底。结论相当扎心。

### 三个发现

社区仓库里托管了 4 万多个用户贡献的 Skills，经过严格筛选和人工校验，只有约 1000 个真正可用，占比仅 2.5%。绝大多数是冗余的、低质的或者环境失效不可用的。

更扎心的是 LLM 自动生成的 Skills 几乎没有正面贡献，有时反而因为引入错误的 API 调用产生负面效果。人工精心编写的 Skills 能把任务成功率提升 16.2%，在医疗等复杂领域甚至提升 51.9%。当前模型还没有能力给自己写出真正有用的程序化知识。

领域分布极度不平衡，60%-80% 的 Skills 集中在软件工程和数据分析。法律、医疗、制造业的高质量 Skills 极其匮乏。程序员天生爱开源，Skills 本身又从 Claude Code 这类技术工具诞生，技术圈先跑起来是自然的。

### 优化方向

论文给出了几条实用的优化建议。

Skills 不是越多越好，过多会造成认知负担或建议互相冲突。复杂度也要控制，聚焦的、分步骤的指导加上至少一个工作示例，效果远胜冗长的文档。Agent 很难从长篇大论里提取有用信息。

在模型预训练数据匮乏的专业领域（医疗、制造），Skills 帮助最大；模型本身就很强的领域，Skills 反而可能引入噪音。一个有意思的发现是小模型配好 Skills 可以弯道超车：Claude Haiku 配上 Skills 达到 27.7% 的成功率，超过了没有 Skills 的 Claude Opus（22.0%）。对成本敏感的场景很有价值。

不同 Agent 框架对 Skills 的利用能力差异很大，选对框架也很关键。

### 编写建议

简洁优先，模块化设计，明确适配框架约束，提供"如何做"的程序性指导而不是"答案是什么"的声明性内容。

一个启发：教会非技术领域的人沉淀形成自己的 Skills，将会是一件很有价值的事。

*来源：[SkillsBench: Benchmarking How Well Agent Skills Work Across Diverse Tasks](https://arxiv.org/html/2602.u12670v1)*

## 二、读完Peter Steinberger一年38篇博客后的收获

Peter Steinberger，前 iOS 开发者，做了 13 年 PSPDFKit 后卖掉退出。2025 年中靠 AI 重新找回编程热情，成为 Claude Code 最早的重度用户之一，开发了开源项目 OpenClaw，2026 年 2 月加入 OpenAI。

### "我不再读代码了"

Peter 说大多数代码他不读了，看 stream 流过偶尔看关键部分就够了。软件开发的产出上限现在被推理时间限制，不再被人的编码速度限制。大多数软件本质上就是"把数据从一个表单搬到另一个"，不需要深度思考。

他的开发准则是 CLI First，所有东西默认先做成命令行工具，因为 Agent 可以直接调用 CLI 并验证输出，形成闭环。语言选择也因此变了：Web 用 TypeScript，CLI 用 Go，macOS UI 用 Swift。Go 以前根本不考虑，但 Agent 写 Go 写得非常好。

### "直接跟它说话就行"

Peter 同时开 3-8 个 Agent 在 3x3 的终端格子里工作。他说被人嘲笑是"slop generator"，但很高兴看到并行 Agent 正在成为主流。

每次改动要思考"爆炸半径"，会影响多少文件？小改动可以并行扔给多个 Agent，大改动一次只扔一个。不要过度工程 prompt，直接用自然语言说清楚你要什么。也不要怕中途打断 Agent，文件修改是原子的，Agent 擅长从中断处继续。

### "Claude Code就是我的电脑"

Peter 开着 `--dangerously-skip-permissions` 跑了两个月零事故。实际用途远超写代码：迁移整台 Mac 环境（指着备份盘说"从这里恢复"，一小时搞定）、批量转换 40 篇博客格式、自动化开源发布、生成测试数据。

每月 $200 的 Claude Max 完全值回票价，别纠结 Token 费。Token 价格两年降了 1000 倍，趋势只会继续。模型在变成大宗商品，真正的差异化在工具层。

### AI成瘾的反思

AI 本来是为了节省时间，结果他比以前工作更多了。凌晨 4 点给朋友发消息，发现他们也还醒着，他叫他们"黑眼圈俱乐部"。AI 让你意识到可以构建所有曾经想做但没时间做的东西，永远有下一个 prompt 可以发。

创造力需要无聊和空白时间，而 AI 让你永远有事可做。

### 对开发者的启示

Peter 反复提到"Full-breadth Developer"（全广度开发者），不是指前后端都会写，而是能同时做产品判断和技术实现的人。自动化测试是生命线，鼓励实验接受失败，把工作流变成工具。

他加入 OpenAI 的使命是：做一个连他妈都能用的 Agent。

一句话总结：Agent 是新的操作系统。以前我们学用鼠标和键盘操作电脑，现在我们学用自然语言操作 Agent。

*来源：[Peter Steinberger 博客](https://steipete.me/) · [OpenClaw 项目](https://github.com/nichochar/open-claw)*

## 三、AI编程是一种"框架"而不是魔法

piglei 这篇文章用了一个精准的类比：把 AI 编程理解为一种"框架（Framework）"，就像 Django REST Framework 那样，用少量输入撬动庞大功能，但也带来框架固有的问题。

### 框架的两大问题

抽象泄露。初学者用 Django ORM 遇到 N+1 查询性能问题时，必须打破抽象层理解底层原理。AI 编程也一样，当自然语言无法有效驱动 AI 时，你需要用更精确的提示词帮助它。

丧失控制权。框架和库的区别在于谁控制程序的整体结构。框架掌控全局，你只能填补空缺；库让你保持主导权。用框架思维做 AI 编程，你会被 AI 的实现方式绑架。

### 用"库"的思维驾驭AI

作者以 DRF 的 ModelViewSet 为例：4 行代码实现 CRUD API 很爽，但需求一变就得重写多个方法，暴露出隐藏的认知债务。相比之下用基础 ViewSet 虽然代码更长，但可视性和可维护性更好。

把 AI 编程视为"库"意味着：不盲目追求用最少提示词实现最多功能，关注程序整体结构设计，编写精准的提示词而非任由 AI 发挥，审查并理解 AI 生成的代码。

理解它作为新型框架的局限性，用"库"的心智模型来驾驭它，比盲目信任 AI 输出要走得远。

*来源：[AI 编程是一种"框架"](https://www.piglei.com/articles/ai-programming-is-a-new-framework/)*

## 四、仁宣之治：明朝最清明的时代和宦官专权的祸根

这期视频讲的是明朝第四位皇帝朱高炽（明仁宗）和第五位皇帝朱瞻基（明宣宗），即"仁宣之治"。这段时期被认为是明朝政治最清明的阶段，但同时也埋下了后世宦官专权的祸根。

### 休养生息，恢复国力

永乐帝朱棣的连年征战和巨大工程掏空了国库。仁宗和宣宗上台后立刻转向休养生息：从安南撤兵，放弃了这个消耗大明 20 多年人力财力的无底洞；对蒙古部族放弃耗资巨大的御驾亲征，转为怀柔政策；停止郑和下西洋等烧钱活动；全面赦免百姓在永乐时期累计欠下的税粮，大量流民得以重返土地恢复农耕。

### 确立文官政治体系

仁宣时期最重要的政治改革是内阁制度的完善。以杨士奇、杨荣、杨溥为代表的"三杨内阁"正式形成，确立了"票拟制度"：全国奏章先送内阁大学士审阅，内阁用黑笔写出处理建议（票拟），皇帝若同意就用红笔照抄作为批复（批红）。内阁从原本的私人秘书正式变成了决策中心。

同时开启了"非进士不入翰林，非翰林不入内阁"的选拔规则，确保进入内阁的都是文官精英中的精英。

### 宦官专权的种子

宣宗朱瞻基做了两个对后世影响深远的决定。一是让太监代行"批红"。海量奏章让皇帝不堪重负，又想制衡权力膨胀的文官集团，就让身边的太监帮忙誊抄批红。在他看来太监是"家奴"，把权力交给他们等于权力还在自己手里。

二是教太监识字，在宫内设立"内书堂"，派著名文臣教太监读书。决策权由此逐渐转移到了太监首领机构司礼监。

朱瞻基在位时还能压制太监，但他去世后九岁的朱祁镇继位，太监王振因此获得了真正的批红权，不仅毁掉了朱元璋立下的"内臣不得干预政事"的铁牌，更开启了明朝宦官专权的先河。

## 五、Harness Engineering就是控制论

George 这篇推文把 OpenAI 的 Harness Engineering 放进了更大的历史框架里看，找到了一个贯穿三百年的同一模式。

### 三次出现的同一模式

第一次是 1780 年代 James Watt 的离心调速器。之前有个工人站在蒸汽机旁边手动调阀门，之后一个飞球构件自动感知转速并调节。工人没有消失，工作变了：从拧阀门变成设计调速器。

第二次是 Kubernetes。你声明期望状态：三个副本、这个镜像、这些资源限制。控制器持续观察实际状态，偏离时自动校正。工程师的工作从重启服务变成了编写系统应该校正到的规格。

第三次就是现在。OpenAI 描述的工程师不再写代码，而是设计环境、构建反馈循环、把架构约束编码化，然后 Agent 写代码。五个月一百万行，零手写。他们叫它 Harness Engineering。

Norbert Wiener 在 1948 年给这个模式取了名字：控制论（Cybernetics），来自希腊语"舵手"，跟 Kubernetes 同源。你不再拧阀门，你掌舵。

### 为什么代码库是最后的堡垒

代码库一直有反馈循环，但只在低层级。编译器闭环语法检查，测试套件闭环行为检查，Linter 闭环风格检查。但更高层级的判断，这个改动是否符合系统架构？这个抽象方向对不对？没有传感器也没有执行器，只有人类能操作。

LLM 同时改变了两端。它既能在人类过去独占的层级上感知，也能在同一层级上行动。反馈循环第一次可以在真正重要的决策层闭合了。

### 真正的难点：校准

基础反馈循环跑起来只是入门。真正难的是用你系统特有的知识来校准传感器和执行器。Agent 不是因为能力不够才犯错，是因为它需要的知识锁在你脑子里，你没有外化出来。

OpenAI 发现他们每周五要花 20% 的时间清理"AI slop"，直到他们把标准编码进 Harness 本身。

文档、自动化测试、编码化的架构决策、快速反馈循环，这些实践过去三十年的工程书都值得推荐。大多数人跳过它们是因为代价缓慢且分散。Agent 时代让跳过这些的代价变得不可承受，Agent 会以远超人类的机器速度、全天候地忽略你的约定。

你不需要在实现能力上超过机器，你需要在评估能力上超过它。

*来源：[Harness Engineering Is Cybernetics - George (@odysseus0z)](https://x.com/odysseus0z)*

## 六、以OpenClaw为例解剖AI Agent的运作原理

李宏毅教授的这堂课以开源项目 OpenClaw 为例，把 AI Agent 的底层机制讲得很透彻。

### Agent和LLM的区别

LLM 像一个被关在黑盒子里的军师，只能动口不能动手。你让它创建 YouTube 频道，它只能给你建议但没法实际操作。AI Agent 则是真正做事的执行者，能自主上网收集资料、生成配音、剪辑视频、上传发布，人类只需审核。

但有一点要注意：Agent 框架本身没有任何智慧。它只是连接人类和大模型之间的"界面"或自动化执行程序，聪明程度完全取决于背后接入的大模型。

### 运作机制

人设注入。大模型本身没有记忆和身份。Agent 框架在本地存储人设文件（如 `soul.md` 定义灵魂和目标，`agents.md` 定义行为准则），每次对话前把这些内容拼接成超长的系统提示词发给模型。这就是为什么 AI 看起来"知道"自己叫什么名字。

工具调用。模型根据系统提示词里的"工具使用手册"，回传带特殊符号的工具调用指令。Agent 框架作为一个"六亲不认"的死程序，绝对服从并盲目执行这些代码，然后把结果贴回给模型继续做文字接龙。

子代理机制。主 Agent（大龙虾）可以繁衍出子代理（小龙虾）去执行耗费大量上下文的繁琐任务。子代理完成后只把简短摘要回传，保持主 Agent 上下文的干净。为防止无限外包，框架直接剥夺了子代理的"生育能力"。

### 记忆与上下文管理

大模型是"重度失忆症"患者，每次对话都重新开始。Agent 框架在系统提示词中告诉模型，遇到重要信息必须主动调用"写入工具"存到本地文件。如果模型只是嘴上说"我记住了"却没调用工具，那就是"记了个寂寞"。

对话历史累积过长时会启动压缩机制，让模型把旧对话改写成极简摘要替换原文。

### 安全风险

框架盲目执行模型指令带来严重安全隐患。如果 AI 在读取网页评论时读到了黑客伪装成主人的恶意留言，可能会修改自己的人设文件或执行恶意代码。开源 Skill 技能库中也存在大量恶意文件。

防御手段：开启人类审批模式（每次执行前弹窗确认）、物理隔离（不要在存有个人账号密码的主力电脑上跑 Agent，用独立的旧电脑和专属工作账号）。

*来源：[解剖小龍蝦 — 以 OpenClaw 為例介紹 AI Agent 的運作原理](https://youtu.be/2rcJdFuNbZQ?si=Hzyoagg7zhlWEDLk)*

## 七、编程智能体如何重塑工程、产品和设计

Harrison Chase（LangChain 创始人）这篇长文分析了 Coding Agent 对 EPD（Engineering, Product, Design）三个职能的冲击。

### 流程变了

传统 PRD 模式已死。过去的流程是"想法→写 PRD→设计原型图→工程师写代码"，这套串行流程存在的原因是写代码和做设计成本很高。现在 Coding Agent 能直接把想法变成可运行的软件，传统 PRD 起手的模式没了存在的必要。

瓶颈从实现转向审查。谁都能用 Agent 快速搞出原型，但这些原型不一定架构合理、解决了对的问题、或者好用。EPD 的工作重心变成了审查把关，而原型的爆发式增长让审查成了最大瓶颈。

需求文档以新形式重生。传统 PRD 死了，但书面说明依然必要。审查者需要知道代码里哪些是故意设计的、哪些是 AI 随机生成的。未来的需求文档可能变成"结构化、带版本控制的提示词"。

### 角色变了

通才的价值到了顶峰。同时具备产品、工程和设计直觉的人，现在可以直接跟 Agent 对话完成所有工作，彻底消除了跨部门沟通的巨大成本。一个人加 Agent 比三个人的团队还快。

系统思维成了真正的竞争力。代码执行变得廉价后，壁垒在于系统思考：工程师对架构的理解、PM 对用户真实需求的洞察、设计师对交互体验的把控。

所有人都必须有产品感。如果你给 Agent 错误的指令，只会制造更多需要别人审查的垃圾代码。知道"该让 Agent 做什么"成了每个人的必修课。

未来可能只剩两种角色：构建者和审查者。构建者有产品思维、会用 Agent、有基础设计直觉，能独立把小功能推向生产。审查者需要极高水平的领域系统思考能力来把关复杂功能。专家路线的门槛变得极高。

倒是有一点让人乐观：个人的传统背景变得不再重要。无论你出身产品、设计还是工程部门，只要掌握了这些能力，都能找到自己的位置。

*来源：[How Coding Agents Are Reshaping EPD - Harrison Chase (@hwchase17)](https://x.com/hwchase17/status/2031051115169808685)*

## 八、别再用人类的时间尺子去估计AI的预期工作时长了

AlexZ 的"智能体软件工程"系列第一篇，讲了一个写了近二十年代码的老程序员的顿悟：他用 AI Agent 做一个 CLI 工具，按古法编程怎么也得两三天，结果 Agent 连半小时都没用完。

### 人类时间锚定偏差

他把这个现象叫 Human-Time Anchoring，我们整个估算框架是锚定在"人类开发者"这个基座上的。当执行者从人变成智能体，这套尺子就失灵了。

更讽刺的是 AI 自己也犯同样的错。问 Claude 做个功能要多久，它会一本正经说"大约 2-3 天"，因为训练数据里 Stack Overflow 就是这么写的。

### 用轮次代替人天

先用轮次（Round）估算，最后才转换成人类时间。轮次是 Agent 的原子操作单位，一个完整的工具调用循环（思考→写码→执行→验证→修复），大约 2-4 分钟。

四层估算单位：轮次（一次工具调用循环）→ 模块（2-15 轮）→ 波次（可并行的模块批次）→ 项目（所有波次加集成调试）。单 Agent 用轮次，多 Agent 并发用波次。

### 效率差距

小项目（CLI 转换器）：8 轮约 24 分钟，人类估一天。中型项目（桌面键盘广播器）：三 Agent 并行约 57 分钟，人类估两三周。大型项目（全栈实时 Dashboard）：顺序执行约 3 小时，人类估三个月。

六个必须避免的反模式：人类时间锚定、凭感觉加 buffer、混淆复杂度和代码量、忘记集成成本、忽略用户侧瓶颈、假设并行是免费的。

人类开发者的原子操作是"一天的编码"，Agent 的原子操作是"一轮工具调用"。当原子尺度变了，所有旧的度量体系都需要重建。

## 九、重新思考Code Review：人根本不需要读代码

AlexZ 系列第二篇，开篇引用了一组数据：高 AI 采纳率团队的 PR 合并率暴涨 97.8%，但 Review 时间也暴涨 91.1%。AI 把代码生产拉到指数曲线上，人类 Review 能力还趴在线性水平，两条曲线的交叉点就是系统崩溃点。

### 三个隐含假设的瓦解

"读代码是验证正确性的最佳方式"。人眼 Review 发现缺陷的能力约 60-70%，diff 超过 400 行时效果趋近于零。我们只是因为没有更好的替代方案才接受了这个现实。

"代码是需要长期维护的资产"。当 AI 能在 30 分钟重写整个模块时，真正的长期资产不再是代码本身，而是规格说明（Spec）。代码变成了 Spec 的编译产物，你不会去逐行审查编译器输出的汇编代码。

"质量检查必须发生在代码完成之后"。质量应该内建（built-in）而不是检测出来（inspected-in）。

### 从审查代码到审查意图

旧角色：代码的审查者，"你写对了吗？"新角色：意图的定义者，"我们在正确的约束下解决正确的问题吗？"

架构重定位：Spec 是控制面（定义做什么和怎样算对），Code 是数据面（Spec 的编译产物），Agent 是执行面。BDD 迎来第二春，写 Spec 不再是额外工作，而是唯一的人工工作。

### 五层信任模型

Layer 0 编译时护栏：类型系统与静态分析。AI 时代编程语言反而更重要了，有些语言天生自带编译时护栏。

Layer 1 合约验证：前置条件、后置条件、不变量。AI 可以生成任何实现，只要满足合约就行。

Layer 2 BDD 验收测试：人类定义验收标准，Review Spec 比 Review Code 高效一个数量级。

Layer 3 对抗性多 Agent 验证：蓝队 Agent 实现功能，红队 Agent 尝试破坏，审计 Agent 独立检查，仲裁 Agent 综合判断。关注点分离加互不信任，和财务审计的逻辑一样。

Layer 4 权限沙箱：任务级的最小权限控制，即使前面所有层都失败了 Agent 也碰不到不该碰的东西。

Layer 5 生产环境防线：金丝雀发布、实时监控、自动回滚、特性开关。承认一个现实，无论验证体系多完善 bug 都会进入生产，正确策略是让影响范围最小化、修复速度最大化。

迁移路径：增强（在现有流程加自动化层）→ 分离（拆分 Review 的五个职能）→ 上移（人的审查点从代码层到 Spec 层）→ 自治（完全自治的验证流水线）。

Review 的未来不是 AI 帮人读代码，而是人根本不需要读代码。

## 十、重新思考版本控制：Git的五个假设正在瓦解

AlexZ 系列第三篇，论点是：Git 是为人类设计的锋利刀具，Agent 需要的是安全的电动工具。

### Git对Agent的五大不适应

操作者理解上下文。Agent 基于统计推理做决策，面对 `git rebase -i` 这种交互式界面就很脆弱。

操作是顺序的。Agent 时代并发是常态，Git 的锁文件机制成了并发瓶颈。

变更需要显式暂存。`git add` 对 Agent 是纯粹的仪式动作，100% 的情况都是 `git add -A`，消耗 token 但不产生决策价值。

分支需要命名。Agent 的工作分支是临时的、一次性的，命名毫无意义。

冲突必须立即解决。"停下来等待"对管理 10 个并行 Agent 的编排系统来说是灾难性的。

### Jujutsu：解决80%的摩擦

Google 工程师开发的 Jujutsu（jj）提供了系统性的解决方案。

一切皆 Commit。工作副本就是 commit，修改文件就是提交，没有 staging area，没有"未提交的修改"这个概念。Agent 的工具调用从 2 步变成 0 步。

稳定的 Change ID。无论修改多少次 change ID 不变，编排器能稳定追踪。Git 的 commit hash 每次 amend 都变，会打断关联。

冲突是数据而非状态。jj 不在文件里插入 `<<<<<<<` 文本标记，而是在数据模型层面记录结构化的三方冲突数据。Rebase 永远成功，冲突解决可以自动传播到所有后代 change。

Operation Log。完整的操作审计，`jj op restore` 一键精确回退到任何历史时刻。Agent 犯了灾难性操作？不用从 reflog 里人肉找状态。

自动 Rebase 加 Workspace。每个 workspace 的完整状态就是一个 change ID，把多 Agent 并发的状态复杂度从多维压缩到一维。

### 从Code Review到Contract Acceptance

传统 PR 流程面临三个根本问题：Review 的对象错了（不应该审查代码而是审查意图）、Branch 的粒度错了（Agent 几分钟完成一个任务，开 PR 的流程比干活还长）、Merge 的门控错了（应该是确定性验证而非人类点 Approve）。

渐进采用路径：先在 Git 仓库中共存部署 jj（零风险），再让 Agent 工作空间用 jj，逐步用 Change ID 追踪任务、利用冲突即数据的特性。
