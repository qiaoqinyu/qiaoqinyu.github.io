## 一、深度学习基础：张量操作与广播机制

今天看了《动手学深度学习》里关于张量操作的部分。虽然是基础内容，但那个广播机制，我之前一直以为很简单，结果实际写代码时没少踩坑。

### 张量是什么

张量就是多维数组。一维是向量，二维是矩阵，三维及以上就叫张量了。PyTorch、TensorFlow这些框架里的张量，看着跟NumPy的数组差不多，但有两个杀手锏：能自动求导，还能用GPU加速。NumPy就只能跑在CPU上。

创建张量的方法很多，可以用`arange`生成序列，用`zeros`或`ones`生成全零全一的，也能从随机分布里采样，或者直接用Python列表来构造。每个张量都有形状，通过`reshape`可以改变形状但不改变数据内容，比如把12个元素的向量变成3行4列的矩阵。用-1可以让系统自己算某个维度应该是多少。

### 索引、切片和运算

索引和切片跟Python列表的用法一样，从0开始，支持负数索引。加减乘除这些运算符都被重载了，变成按元素逐个计算。两个形状一样的张量做运算，就是对应位置算一下。还能做逻辑比较，比如`X == Y`会得到一个布尔张量，也可以对整个张量求和。

### 广播机制

这部分容易出错。两个张量形状不完全一样时，只要满足特定条件，系统就能自动把它们“撑开”成一样的形状再做计算。规则是从后往前对齐维度，要求对应维度要么相等，要么其中一个为1，要么其中一个压根不存在。系统会沿着长度为1的维度复制数据，让两个张量能对上号。

这让我想到小时候玩拼图，有时候差一小块，就得把旁边那块复制一份填进去。

### 内存管理

深度学习里参数动不动就上亿，还老要更新。如果每次做`Y = X + Y`这样的操作都开一块新内存，开销太大，还可能让其他引用指向过时的数据。最好用原地操作，比如`Y[:] = X + Y`或者`X += Y`，直接复用原来的内存。

PyTorch和NumPy互相转换时，底层数据通常是共享的，改一个另一个也跟着变。MXNet转换后就不共享了，据说是为了防止NumPy阻塞GPU计算。如果张量里只有一个元素，可以用`.item()`把它转成普通的Python数字。

*来源：[Data Manipulation — Dive into Deep Learning 1.0.3](https://d2l.ai/chapter_preliminaries/ndarray.html#broadcasting)*

## 二、Agent Harness 架构解剖

LangChain的Viv写了篇长文，把Agent Harness讲得很透。他给了一个干净的定义：智能体 = 模型 + 缰绳（Harness）。模型提供智力，Harness负责把智力变成可用的行动。所有不是模型本身的东西，代码、配置、执行逻辑，都算在Harness里。

### Harness 的五层结构

一个光秃秃的模型算不上智能体。它只能接收文本图片然后输出文本，没法记住之前聊过什么，没法执行代码，也没法自己搭环境。Harness就是来补这些能力的。具体分五层：

- 系统提示词
- 工具、技能、MCP及其描述
- 捆绑的基础设施（文件系统、沙箱、浏览器）
- 编排逻辑（生成子智能体、交接任务、路由到不同模型）
- 确保确定执行的钩子和中间件（压缩上下文、强制继续、代码检查）

### 从期望行为反推 Harness 设计

文件系统是最基础的Harness组件。智能体需要持久存储来跟真实数据打交道，把记不住的东西存下来，在不同会话间保留工作进度。加上Git就有了版本控制，能追踪修改、回滚错误、开实验分支。

与其给智能体预设一堆固定工具，不如直接给它一个Bash终端。模型可以在运行中自己写代码来创造工具，不受预配置的限制。沙箱提供安全的隔离环境，里面预装了语言运行时、Git、测试工具和浏览器。

记忆方面，Harness支持像AGENTS.md这样的记忆文件标准，启动时自动加载。配合Web Search和Context7这类工具，智能体就能获取训练数据截止日期之后的新信息。

### 对抗上下文腐烂

Context Rot指的是上下文窗口被塞满后，模型推理能力会明显下降。Harness有三个办法应对：压缩（把现有上下文智能地卸载和总结）、把工具调用输出卸载掉（减少大型工具输出的噪音）、技能的渐进式披露（避免一启动就把所有工具描述都灌进去）。

### Ralph Loop 与长周期执行

让智能体自主编写软件是终极目标。Ralph Loop是一种钩子机制，当模型试图说“我干完了”要退出时，拦截它，在一个干净的上下文窗口里重新注入原始任务提示，强迫它继续工作直到真正完成。

### 模型训练与 Harness 的共同进化

Claude Code和Codex这些产品，在后训练阶段已经把模型和Harness放在一个闭环里一起进化了。好用的组件被发现后就加入Harness，然后用来训练下一代模型。但这种共同进化会影响模型的泛化能力。Terminal Bench 2.0排行榜上有个有趣的现象：Opus 4.6在Claude Code里的得分，远低于它在其他Harness里的表现。

未来还有不少开放问题：怎么在共享代码库上协调几百个并行智能体，怎么让智能体分析自己的运行记录来修复Harness层面的故障，怎么开发能动态组装工具和上下文的Harness系统。

*来源：[The Anatomy of an Agent Harness - Viv](https://x.com/Vtrivedy10/status/2031408954517971368)*

## 三、用 Harness 工程把编程智能体从 Top 30 拉到 Top 5

还是Viv的文章，这篇更偏实战。他们的编程智能体deepagents-cli在Terminal Bench 2.0上从52.8分提到了66.5分，排名从前30冲进了前5。整个过程只改了Harness，模型（gpt-5.2-codex）一点没动。

### 实验设置

评测用的是Terminal Bench 2.0，包含89个跨机器学习、调试和生物学的任务。用Harbor调度Daytona沙箱，LangSmith做运行追踪。优化主要调了三个地方：系统提示词、工具、以及模型和工具调用前后的钩子（中间件）。

### Trace 分析器

他们搞了一套类似boosting算法的错误分析流程：从LangSmith拉实验的运行记录，并行生成多个错误分析智能体，汇总反馈后做针对性修改。每一轮都专注于解决上一轮的错误，人类可以在第三步介入，防止过拟合到测试集上。

### 四大改进策略

第一，构建与自我验证。最常见的失败模式是智能体写完代码后，自己重读一遍觉得没问题就停了。他们引入了明确的工作流：先规划与发现，然后构建，接着验证，最后修复。用了一个叫PreCompletionChecklistMiddleware的钩子，在智能体即将退出前拦截它，提醒它必须实际运行一遍代码来验证。

第二，给智能体环境上下文。用LocalContextMiddleware在启动时把当前工作目录和文件夹结构映射给智能体，帮它了解环境。还教智能体写可测试的代码，并通过时间预算警告来督促它按时完成。

第三，鼓励退一步思考。LoopDetectionMiddleware会跟踪对单个文件的编辑次数，编辑太多次后就主动提示“考虑重新评估你的方法”，防止它钻牛角尖。

第四，合理分配推理算力。gpt-5.2-codex有四种推理模式（低/中/高/极高）。他们用了“极高-高-极高”的“推理三明治”策略，在规划和验证阶段分配更多算力。全用极高模式只拿了53.9%，因为超时太多了。

### 核心经验

- 主动替智能体做上下文工程，别让它自己瞎琢磨。
- 帮智能体自我验证，模型倾向于接受自己的第一个方案。
- 把运行追踪当成反馈信号来用。
- 为不同模型定制Harness，Claude用早期框架只得了59.6%，因为没跑针对它的优化循环。

*来源：[Improving Deep Agents with Harness Engineering - Viv](https://x.com/Vtrivedy10/status/2023805578561060992)*

## 四、Pandas 加载数据到底能撑多大

今天看到一个关于pandas `read_csv` 加载能力的分析，从四个维度拆解了实际限制。这问题我确实遇到过，内存直接炸了。

### 内存是最硬的天花板

经验是，内存需求大概是数据文件大小的2到5倍。CSV是纯文本，转成DataFrame要经过字符串解析、类型推断、内存复制，这个过程会让内存占用膨胀2到10倍。一个1GB的CSV，在内存里可能占掉3到8GB。

普通笔记本（8-16GB内存）：安全范围是100MB到1GB的CSV，极限大概2-3GB，超过就可能内存不足或者系统卡死。服务器（64GB以上内存）：能加载10GB到50GB以上，但仍然受CPU、IO和pandas单线程读取的限制。

### 读取时间

磁盘速度（SSD比HDD快很多）、文件大小、是否压缩（gzip会更慢）都影响读取速度。小于100MB几秒搞定，大于5GB可能要好几分钟。pandas读CSV是单线程的，扩展性天生有限。

### 处理开销

加载后做`groupby`、`apply`、one-hot编码这些操作，时间复杂度高，还会产生额外的内存副本，进一步加剧内存压力。

### 改进方案

分块读取：用`pd.read_csv('data.csv', chunksize=100000)`逐块处理。优化数据类型：用`dtype={'col': 'int32'}`显式指定更小的类型。改用二进制格式：比如parquet或feather，读写更快更省内存。换工具：试试polars（更快更省内存）、dask（分布式）、spark（处理大数据）。

*来源：[ChatGPT 对话 - Pandas 数据加载规模分析](https://chatgpt.com/s/t_69b973aaf1dc8191bd16f213b9ed8cba)*

## 五、Agents in the Outer Loop：把智能体部署到云端

OpenHands这篇博客聊了AI辅助开发的一个新方向：从“内部循环”扩展到“外部循环”。这想法挺有意思，把智能体从副驾驶升级成了云端自动化工人。

### 内部循环 vs 外部循环

内部循环是开发者在本地写代码的过程，改代码、跑测试、反复迭代。这是高度个性化的环节，你爱用什么IDE就用什么。外部循环是`git push`之后在云端发生的事，CI/CD流水线、代码审查、问题追踪。这是团队协作的地方，工具必须统一。

### 内部循环智能体的天花板

现在大多数AI编程助手都在内部循环里当“结对程序员”，能在写代码阶段带来40%到50%的效率提升。但这种提升是渐进式的，开发者还得全程盯着，离“完全自主的AI开发者”还差得远。有时候反而增加了调试时间。

### 外部循环的好处

安全性方面，本地跑智能体风险大，可能误删系统文件或者窃取AWS密钥，需要开发者频繁手动确认。在云端隔离环境里，智能体的“破坏范围”很小，只拥有被明确授予的权限，可以完全自主运行，不用人看着。

可扩展性方面，本地最多同时跑2-3个智能体，资源争抢加文件冲突，人也看不过来。云端可以轻松扩展到几十甚至几千个智能体同时工作，每个都有专属的虚拟机、内存和CPU。

### 最适合的场景：技术债务和重复劳动

自动修复安全漏洞：一旦有新漏洞公布，自动触发智能体研究、修复、验证并生成合并请求。大公司可以同时启动成百上千个智能体，一次性处理所有受影响的代码仓库。

清理生产环境错误日志：设置智能体扫描Datadog这类系统的错误日志，自动定位触发错误的代码库和具体位置，提出修复方案。凌晨3点出的错，第二天早上就有准备好的修复方案等着你。

### 两者兼备

内部循环智能体适合处理特定的、临时起意的工作，比如实现新功能、排查复杂Bug。外部循环智能体适合处理可重复的、机械性的工作，比如通过自动化流程清理维护积压任务。最好的做法是让每个开发者自由选择内部循环的AI工具，但在外部循环的自动化上采用统一的团队方案。

*来源：[Agents in the Outer Loop - OpenHands](https://openhands.dev/blog/20251202-agents-in-the-outer-loop)*

## 六、Claude Code Skills 使用经验：Anthropic 内部的实战总结

Anthropic的Thariq写了一篇Claude Code内部使用Skills的经验。Skills已经成为Claude Code最常用的扩展点之一，他们内部有几百个在活跃使用。

### Skills 不只是 Markdown 文件

很多人以为Skills就是个Markdown文件。其实Skills是一个文件夹，可以包含脚本、资产、数据等等，智能体能发现、探索和操作这些内容。Skills还有丰富的配置选项，包括注册动态钩子。

### 九大类型

1.  库与API参考：解释怎么正确使用特定库、命令行工具或SDK，包含参考代码和避坑指南。
2.  产品验证：描述怎么测试代码，搭配Playwright、tmux等工具，甚至可以让Claude录制测试视频。
3.  数据获取与分析：连接数据和监控系统（比如Grafana），包含登录凭证和仪表盘ID。
4.  业务流程自动化：把重复工作流（比如站会报告、创建工单）转成单个命令。
5.  代码脚手架与模板：生成框架样板文件，适合那些超出纯代码生成的自然语言需求场景。
6.  代码质量与审查：强制执行代码标准，可以在钩子或GitHub Actions中自动运行。比如有个叫adversarial-review的，会生成一个“用新眼光看问题”的子智能体来挑刺。
7.  CI/CD与部署：比如babysit-pr可以监控合并请求、重试失败的持续集成、解决合并冲突、启用自动合并。
8.  运行手册：根据错误症状，走多工具调查流程，生成结构化的排查报告。
9.  基础设施运维：执行常规维护，为破坏性操作提供安全防护。

### 写好 Skills 的要领

不要陈述显而易见的东西，重点放在能让Claude跳出常规思维的特殊信息上。建立一个“避坑指南”部分，这是信息密度最高的内容。利用文件系统做渐进式披露，把详细文档拆到独立的参考目录里，避免一次性灌入太多信息。

不要给Claude设死板的执行路径，要给它根据具体情况灵活调整的空间。考虑初始化设置，在config.json里存储配置信息。Description字段是给模型看的，Claude扫描它来决定是否有对应的Skill可用。

可以在Skills内部存储数据（日志文件、JSON、SQLite），用`CLAUDE_PLUGIN_DATA`做稳定的持久化存储。给Claude提供现成的脚本和库，让它把精力花在高价值的代码组合上，而不是从头写起。Skills可以包含按需激活的钩子，比如`/careful`阻止危险命令，`/freeze`阻止修改特定目录外的文件。

### 分发与管理

小团队直接提交到代码库里（`./.claude/skills`）。规模大了就用插件市场，让团队自己决定安装哪些。发布前要做有机的筛选整理，避免库里充斥冗余或低质量的Skills。Skills之间可以通过互相引用名称来组合调用。用PreToolUse钩子记录调用日志，来测量各个Skill的使用情况。

*来源：[Lessons from Building Claude Code: How We Use Skills - Thariq](https://x.com/trq212/status/2033949937936085378)*

## 七、Ben Tossell 的开发者周记：工具栈与 AGENTS.md 配置

投资人兼前创业者Ben Tossell在他的邮件通讯里分享了这周的开发记录和他的技术栈。这种实战记录看着很过瘾，能偷师不少具体技巧。

### 本周项目

交互式开发指南：上周办了一场1300人报名的研讨会，结果Codex演示时崩了。为了弥补，他做了一份交互式指南（通过gists.sh生成链接），用户可以直接喂给AI智能体，智能体会一步步指导用户从零构建和部署网站。

可视化技能插件：逆向工程了Claude最新发布的可视化功能，包装成任何AI智能体都能安装的GitHub开源Skill，拿了200多个star，能大幅提升Codex的界面设计表现。

### 多模型混合使用策略

Ben总是混用不同模型。GPT 5.4 XHigh专门用来写“核心代码”、实现新功能和新想法。Opus 4.6则永远用来做项目规划、研究、非技术任务和界面设计。这思路很实用，让合适的模型干合适的事。

### 命令行工具偏好

他极力推荐Droid（能自主运行数小时实现端到端任务构建，Ben投资了这家公司）和Pi（速度极快且轻量，极度遵从用户指令）。主要用Cmux终端应用（内置Ghostty和浏览器，支持面板拖拽管理），配合Zed查看本地文件。智能体客户端最常用Codex app，认为用户体验好且容易上手。

### Skills 和插件

frontend-design（Anthropic出品）：好用，但更期待后续的ui.sh。json-render（Vercel出品）：极速生成UI界面，Ben用它来可视化Mac Mini上的zapier/n8n自动化流程。agent-browser（Vercel出品）：后台开Chrome浏览器像真人一样点击导航、截图检查错误，通过dogfood标签拦截错误并生成修复报告。react-doctor：自动检查和修复AI生成的React代码。

### AGENTS.md 配置哲学

Ben的AGENTS.md（或符号链接的CLAUDE.md）每次会话开始时自动加载。他设定的开发循环包含几个严格步骤：

强制建立规范的文件夹架构，要求AI创建`/spec/`文件夹，以`00_spec1.md`的数字递增格式保存规划文件。强制记录开发进度到`progress.md`，解决AI上下文记忆被压缩或丢失的问题。自动化测试与自纠错，要求AI在把测试链接发给他之前，必须先用agent-browser配合dogfood标签自行跑一遍浏览器测试。强制查阅官方文档，防止AI依赖过时的自有知识。

一个有趣的验证环节：为了确认AI确实加载了指令，他要求AI会话第一句话必须回复电影《冰上轻驰》的一句台词。这招挺聪明，像个暗号。

### 指令漂移问题

评论区有开发者提出了“指令漂移”问题。随着配置文件不断更新，如果出现前后矛盾的指令（比如一段要求AI自主运行，另一段要求执行前必须确认），AI系统的可靠性会急剧下降。这类系统文件需要定期维护和清理，不能光往里加东西。
