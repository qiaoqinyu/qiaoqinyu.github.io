## 一、OpenAI Codex 最佳实践指南

今天看了OpenAI官方出的Codex最佳实践。看完感觉，他们想说的核心就一句话：别把Codex当成一个用完就扔的代码生成器，你得把它当成一个能长期合作、可以不断调教的“新同事”。

### 好的提示词长什么样

一个能用的提示词，得说清楚四件事：你想干啥（目标），这事跟哪些文件有关（上下文，可以用@引用），有什么规矩不能破（约束条件），以及干到什么程度算完（完成标准，比如测试通过）。任务越难，你就得让Codex想得越深，把推理级别调高。

### 复杂任务先做计划

碰到那种模模糊糊或者特别复杂的活儿，别急着让它写代码。先用`/plan`或者按`Shift +Tab`切到计划模式，让它先理理思路，做个方案。它甚至能反过来问你问题，帮你把模糊的想法问清楚。要是那种需要好几天、分好多步的大项目，最好准备一个`PLANS.md`模板，让它照着模板一步步来。

这点让我挺意外，我之前总觉得AI就是直接出结果，没想到计划阶段这么重要。

### 用 AGENTS.md 固化规范

如果你发现某个提示词套路特别好用，每次都管用，那就别每次都手动敲一遍了。把它写进一个叫`AGENTS.md`的文件里。这个文件就像是给Agent看的“员工手册”，里面写清楚代码库长啥样、怎么构建和测试、团队有啥规矩、哪些事绝对不能做、以及“完成”到底是什么意思。这个文件可以放好几个地方，比如全局目录、项目根目录或者子目录里，离当前工作目录越近的，优先级越高。手册别写太啰嗦，只在Codex反复犯同一个错误时，才往里加新规则。

这让我想到，就像每次跟新同事合作，你不可能每次都从头讲一遍公司规章，有个手册就方便多了。

### 测试和审查形成闭环

别光让Codex写代码，得让它参与到完整的验证流程里来。让它写测试，跑Lint检查，确认代码行为符合预期。用`/review`命令让它做类似代码审查的活儿，也可以配合一个`code_review.md`文件来统一团队的审查标准。甚至可以把Codex配置成能自动审查GitHub上PR的“机器人”。

### MCP 接入外部上下文

有些信息不在代码库里，比如经常变动的数据源，这时候可以用MCP（模型上下文协议）去连接外部系统。通过命令行`codex mcp add`就能添加。不过别一上来就把所有工具都连上，先从一两个最能帮你省事的开始。

### Skills 和 Automations

那些重复性的工作，可以封装成`SKILL.md`文件，定义好输入、输出和触发词。等这个Skill足够稳定了，就可以通过Automations设定一个时间表，让它在后台（比如一个独立的Git工作区里）定期自动运行。比如让它定期总结提交记录，或者扫描漏洞。简单说，Skills管“怎么做”，Automations管“什么时候做”。

### 会话管理

记住“一个任务一个线程”，只有工作真的出现分支了，才用`/fork`。线程太长了就用`/compact`压缩一下，用`/resume`恢复之前的会话。主要工作放在主线程，测试或者探索性的任务，可以交给子Agent去并行处理。

### 常见坑

文档里也列了几个容易踩的坑：把长期有效的规则写在一次性的提示词里，而不是`AGENTS.md`里；不给Agent提供构建和测试命令，导致它没法自己验证代码；复杂任务跳过计划阶段；在不了解工作流的情况下就给Agent完全权限；还有，把一个项目的所有任务都塞进同一个线程，搞得上下文又乱又长。

*来源：[Best practices - Codex](https://developers.openai.com/codex/learn/best-practices)*

## 二、文件系统并非你所需的全部

Kevin Gu在X上发了篇长文，回应最近挺火的一个观点，就是“用Markdown文件来构建企业知识图谱”。他的结论很直接：这是在用最差劲的介质，重新发明一遍数据库。

### 上下文确实是瓶颈

企业里的知识，像各种决策、会议记录、优先级变化，确实散落在各处。Agent每次启动都得从头了解，这个诊断没错。但解决办法，不该是把所有东西都塞进Markdown文件里。

### Markdown 的查询能力约等于 grep

你建了一个复杂的文件夹树，按部门、项目、决策分门别类，恭喜你，你其实设计了一个数据库的表结构（Schema）。但当你真想查点东西，比如“所有跟当前路线图有矛盾的决策”时，你只能靠全文搜索（grep）。真正的数据库，天生就有关联查询、索引和约束这些能力。

### 维护才是致命问题

有人说“Agent不会厌倦维护”，但问题出在结构上。你从Google Doc里把信息提取到Markdown笔记里，瞬间就有了两份副本。只要有人改了原始文档，你的笔记就过时了。而数据库的做法是只存一个指向源头的引用（比如Drive文件ID、Slack消息链接），信息之间的依赖关系编码在数据库里，源头一更新，所有相关的东西都能跟着变。保证信息不漂移，靠的是架构设计，不是靠人24小时盯着去维护。

### 代码类比在企业上下文中失效

代码是可以执行的，你改了代码，跑一遍测试就能发现错误。但企业上下文不是这样。一篇会议记录里提到了某个战略文档，这不等于代码里的模块导入。产品计划不会因为销售团队有不同的假设就“抛出异常”。矛盾可以安安静静地躺在几段文字里好几个月，读起来还特别通顺。Agent能在你的笔记网络里流畅地跳转，但它分不清哪些内容是已经生效的、哪些是过时的、哪些是有约束力的正式决定，哪些又只是某个人的主观解读。

这个点说得太对了。我之前没细想过，总以为把文档连起来就万事大吉了。

### 扁平化的治理困境

在Markdown文件系统里，一个正式决策、一个随口猜测、一份会议摘要、一个三个月前已经过时的想法，全都被压平成文本块。它们看起来结构差不多，但权威性天差地别。讽刺的是，为了弥补这一点，人们开始在文件系统里加各种元数据、状态标签、类型化链接……这不就是在文件夹树里，偷偷摸摸又重建了一个数据库吗？

### 三层架构才是正解

Kevin Gu提出了一个三层结构。最底层是工作应用层，像Drive、Slack、CRM，这是真实数据的源头。中间是文件层，放指令、标准操作流程（SOP）和Agent的草稿本，这相当于热访问的元数据，用来启动Agent并告诉它去哪找东西。最上面是数据库层，这里存放提取出来的结构化信息、实体之间的关系、溯源指针和依赖链条，这才是真正的、可查询的知识图谱。

### 给你的 Agent 一个数据库

理想情况下，Agent应该能自己写数据处理的流程（ETL），通过API从工作应用里抓取数据，维护一个带有类型化关系和依赖追踪的结构化索引。源头更新了，它知道该重新评估哪些部分。你否决了它的一个建议，这条记录会被记下来，它就能随时间不断学习。未来企业之间比的，不是谁的笔记写得漂亮，而是谁建起了一套结构化的、能追溯来源的上下文基础设施。想让Agent从副驾驶变成真正的数字同事，光靠Markdown文件可不够。

*来源：[Kevin Gu - A File System Is Not All You Need](https://x.com/gukevinr/status/2031889622385729730)*

## 三、AI 原生时代怎么招工程师

Augment Code团队写了篇文章，讨论当AI Agent能写99%的代码时，招工程师到底该看什么。他们的结论是：“写代码”这个能力已经不能作为区分标准了，现在要看的是“判断力”。

### 角色在变

工程师干的事变了：以前是亲手写代码，现在是定义意图、评估各种选择的利弊。以前是自己实现方案，现在是编排AI Agent去干活。关注点也不一样了，不再是“怎么解决这个问题”，而是“该不该解决这个问题”。衡量产出的标准，自然也就从个人写了多少行代码，变成了整个系统取得了什么结果。

### 六大能力维度

他们总结了AI原生工程师需要的六项能力：

第一是产品与成果品味。我们正在做的东西是对的吗？代码生成变得极其便宜之后，做错产品就成了最大的浪费。最高效的工程师不再是写代码最多的那个，而是能拨开迷雾、锁定正确问题的人。

第二是系统与架构判断力。这东西能在生产环境里活下来吗？AI生成的代码也许能跑通，但它缺乏对长期权衡、运维现实和规模化风险的判断。“跑得通”很容易，“持续稳定运行”极难。

第三是Agent杠杆能力。你能把AI转化为真正的工程产出吗？不是把AI当个助手，而是像委派任务给真人一样去管理它。要懂得怎么给AI拆解问题、在它跑偏时把它拉回来、严格地验证它的产出。心里必须清楚，AI效率极高，但偶尔会非常自信地犯错误。

第四是沟通与协作。代码实施速度大幅提升后，工作的重心就转移到了澄清问题、展示各种选择的利弊、整合各方意见上。最快的团队不是写代码最快的，而是能最快达成共识的。

第五是所有权与领导力。对端到端的最终成果负责，主动去解决所有阻碍团队达成目标的障碍，哪怕这些障碍超出了你平时的职责范围。

第六是学习速度与实验心态。现在的工具可能三个月后就过时了。需要持续实验，随时准备改变工作流，更好的方法一出现，就果断抛弃旧的。实验不是一个阶段，它就是日常工作本身。

### 四类人才画像

基于这些维度，他们锁定了四类要招的人：AI原生系统工程师（架构判断力强）、AI原生产品工程师（产品品味和用户同理心强）、AI原生应用AI工程师（深入理解AI模型本身）、AI原生早期专业人员（学习速度优先的新生代）。

*来源：[How we hire AI-native engineers now - Augment Code](https://www.augmentcode.com/blog/how-we-hire-ai-native-engineers-now)*

## 四、AI 原生转型最难的不是技术

同样来自Augment Code，工程副总裁Vinay Perneti写的这篇文章，谈的不是工具和模型，而是转型过程中“人”的问题。

### 什么才算真正的 AI 原生

在代码编辑器旁边开个聊天窗口，让AI帮你写个函数——那只能叫“AI辅助”。真正的AI原生，是把Agent放到工作执行的核心位置，同时跑多个Agent，而人的精力则集中在定义意图、设计和质量控制上。工程师的角色从“作者”变成了“架构师和编辑”。

### 资深工程师的身份危机

对于经验丰富的工程师来说，写代码是他们的身份认同和职业骄傲。当被告知这部分核心工作将被自动化时，感觉像是一种“降级”，从“骄傲的建设者”变成“协调员”，同时伴随着对自身技能萎缩的真实恐惧。

但事情没那么简单。AI原生其实需要更深层次的工程素养。资深工程师积累下来的那些“避坑”经验、模式识别能力和架构判断力，恰恰是统筹和管理Agent最需要的东西。他们的专业知识，其实比以往任何时候都更值钱了。

### 初级工程师的意外优势

初级工程师因为没有多年手动写代码形成的肌肉记忆，反而更敢把那些极具挑战性的任务直接丢给Agent去尝试。新老两种工程师在这个新时代，其实是高度互补的。

### Augment 的“慢下来才能快起来”

他们团队当时做了一件事：停下繁忙的开发节奏，抽出整整两天时间，让工程、产品、设计和研究团队坐在一起做深度对齐。

第一天搞了个黑客马拉松，强制采用“Agent优先”模式——要求人类不能自己动手写代码。这么做的目的是测试Agent的极限在哪里，同时暴露出基础设施的真实差距。

第二天直面情绪。他们通过匿名投票生成词云，结果显示，团队不仅感到“好奇、兴奋、充满希望”，同时也抱有强烈的“挫败感”和“不确定性”。他们意识到，必须公开承认这些复杂的情绪，否则它们就会转入地下，变成消极怠工。

### 团队共创得出了什么

在工具层面，他们发现代码验证是最大的瓶颈。Agent写代码不是问题，问题是它们没法验证代码是否真的能跑（缺乏CI反馈、测试结果或者UI交互能力）。Agent也不了解那些没有记录在案的系统架构和团队内部约定，写出的代码往往是“局部看正确，但放到全局就不连贯”。

在人的层面，工作瓶颈已经从“写代码”转移到了“审查代码”。Agent动辄就生成上千行的PR，审查者的认知负担暴增。规格说明（Spec）成了人类工作中杠杆率最高的部分。需求要是写错了，Agent只会以更快的速度，构建出错误的东西。所以工作流不应该再是“从工单直接到PR”，而必须是“从工单到Spec，再从Spec到PR”。

### 接下来怎么做

他们决定成立专门的Agent生产力团队，来打破验证瓶颈；全面转向“Spec优先”的工作模式；用AI工具来解决AI带来的代码审查负担；投资建设Agent的上下文自主权（比如完善`AGENTS.md`、建技能库），把团队的“部落知识”转化成Agent能理解和消化的内容。

说到底，Agent就是个放大器。你的规格写得好、代码库干净，AI产出的质量就高。如果你的需求模糊、代码库一团糟，AI只会帮你更快地制造出一场灾难。
