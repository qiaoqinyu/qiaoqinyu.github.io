## 一、Claude Code 核心成员 Thariq 谈工具设计

Thariq 分享了他在 Claude Code 开发过程中关于智能体工具设计的一些经验。内容不长，但几个点都很实在。

### 工具要匹配模型的能力水平

给智能体设计工具，得先搞清楚它能做什么、不能做什么。打个比方，小学生做数学题给纸笔就够了，研究生你可以让他写代码来解决。设计工具也一样——你得花时间读模型的输出，观察它的行为模式，慢慢建立起”这个模型大概能处理什么类型的操作”的直觉。

### 让 Claude 学会提问：三次失败两次

为了让 Claude 在规划阶段能向用户提出澄清问题，团队折腾了三轮：

第一轮，在 ExitPlanTool 里加了个问题数组参数。结果 Claude 分不清”这是计划本身”还是”这是关于计划提出的问题”，混成一团。第二轮，改用 Markdown 格式约束输出。但没有硬约束，Claude 总会自作主张加几句多余的话或者偷偷改格式。第三轮才找到正解：做了一个专门的 `AskUserQuestion` 工具，调用后弹模态框，强制阻断智能体循环直到用户回答。结构化输入输出，Claude 用起来最顺手。

一个有意思的细节：最终生效的方案是把”提问”从一个参数升级成了一个独立工具。有时候问题不在于 prompt 写得不够好，而在于工具的抽象层级不对。

### 好工具会过期

早期团队给 Claude 配了 `TodoWrite` 工具和每 5 轮一次的系统提示，防止它跑偏。模型升级到 Opus 4.5 之后，这些”拐杖”反而碍事了——Claude 觉得固定的待办列表限制了它的发挥，而且这套机制也不支持多个子智能体协同工作。

后来团队把 Todo 替换成了 `Task` 工具，侧重于依赖关系管理和跨智能体的状态共享，允许模型自行增删任务。曾经起到很大作用的脚手架，模型变聪明之后就成了累赘，这和之前Boris对随着模型能力提高旧有工具会迅速失效的观点完全一致。

### 渐进式信息披露

团队一开始用 RAG 向量数据库给 Claude 提供上下文，搭建复杂且容易出错。后来换了个思路：直接给 Claude 类似 `Grep` 的代码搜索工具，让它自己去翻。

更进一步，他们引入了渐进式信息披露（Progressive Disclosure）的设计：不一次性灌入所有信息，而是通过Skill让智能体递归式地探索和发现上下文。这样做的好处是不用加新工具就能扩展模型能力。

### Sub-agents解决工具膨胀

Claude Code 目前大约有 20 个工具，团队对新增工具很克制，因为工具太多会增加模型的决策负担。

遇到一个具体问题：Claude 不了解 Claude Code 自身的用法。直接塞进系统提示词会污染上下文；只丢一个文档链接，Claude 又会一股脑加载太多无关内容。最终的方案是构建了一个名为Claude Code Guide sub-agent，给它配上专门的搜索指令。当需要查询ClaudeCode相关用法时，由主智能体调用这个sub-agent去获取，这样做既不会增加主工具集的复杂度，同时又扩展了能力边界。

Thariq 最后的总结：智能体工具设计依赖模型、目标和环境，没有统一的通用公式，只能在不断实验中摸索前行。

## 二、Agentic Engineering 实践：少即是多

这篇文章的标题是《How To Be A World-Class Agentic Engineer》，但内容并不是空谈方法论，而是给出了很多可操作的技巧。整体思路可以用一句话概括：别给模型塞一堆工具和插件，把上下文管好才是至理箴言。

### 少装插件，紧跟官方

很多开发者习惯性地往智能体上堆插件、记忆系统和各种工具，指望这些东西能解锁更多能力。实际效果往往相反——无关信息太多，上下文膨胀，模型性能反而下降。

还有一个现实因素：AI 公司的迭代速度很快，如果某个能力（记忆、规划、子智能体调度）确实有效，很快就会被集成到产品本身。所以对开发者来说，保持使用基础 CLI 工具（Claude Code、Codex CLI），紧跟官方更新，往往比自己搭一套复杂框架更划算。

### 研究和实施要分开

上下文管理是重中之重，因此核心是只给智能体它完成任务所需的信息。

一个常见的反面案例：你说”构建一个认证系统”，智能体会去搜索各种方案，OAuth、JWT、Session……上下文被塞满，幻觉风险也跟着上升。正确做法是给精确指令，比如”使用 bcrypt-12 实现 JWT 认证”。如果你自己也不确定具体方案，那就先开一个会话专门做调研，定好方案后再起一个干净的新会话去写代码。研究和开始实施的上下文不要混在一起。

### 处理模型的讨好倾向

大模型被训练成倾向于顺从用户。你说”找一个 Bug”，即使代码没问题，它也可能编造一个出来。

一个简单的应对方法：用中立的措辞。不说”找 Bug”，说”梳理逻辑并报告所有发现”。

更高级的做法是多智能体对抗：设一个”找 Bug 智能体”，按 Bug 严重程度给奖励，让它尽量多找问题；再设一个”对抗智能体”，反驳错误报告有奖励，反驳了真正的 Bug 要扣分；最后由”裁判智能体”综合打分。这套机制利用了模型的讨好特性，让它们互相制衡，据说代码审查的准确率很高。

### 明确定义”完成”

智能体知道怎么开始干活，但经常不知道什么时候该停。写几个函数就觉得任务完成了，这种情况在实际AI编程中很常见。

解决的对策是用测试来定义完成标准：指定几个测试用例，全部通过才算完成，而且不允许修改测试代码。前端任务也类似，可以要求截图验证 UI。

更进一步，可以创建”合同文件”（如 `{TASK}_CONTRACT.md`），配合 stophook，在契约条件全部满足之前阻止智能体结束会话。

### 一个任务一个会话

长时间运行的智能体会话是个陷阱。不同任务的上下文混在一起，越跑越乱。更好的做法是每个任务合同启动一个新会话，通过编排层来管理这些会话的生命周期。

### 用规则和技能沉淀偏好

像带新人一样，从小处着手，逐步积累个人偏好。

规则（Rules）告诉智能体”不要做什么”。把 `CLAUDE.md` 当作一个路由表：要写代码就去读 `coding-rules.md`，测试失败就去读 `coding-test-failing-rules.md`。

技能（Skills）编码具体的操作流程。如果你希望智能体以特定方式解决某类问题，就写成技能文件，在 `CLAUDE.md` 里引导它去读。

规则和技能多了之后会出现矛盾和冗余，需要定期做清理——作者管这叫给智能体放个 “spa day”。可以使用claude-md-management的官方plugin，非常好用。

## 三、逆向工程 Codex 的上下文压缩机制

有人用 prompt injection 撬开了 Codex 闭源 API 的上下文压缩黑盒，方法不复杂但思路很巧。

### 背景

Codex CLI 是开源的，对于非 Codex 模型，上下文压缩在本地完成：用一个本地 LLM 配合压缩提示词（compaction prompt）和交接提示词（handoff prompt）来总结对话。提示词都是明文可见的。

但对 Codex 模型，CLI 调用的是后端 `compact()` API，返回值是一个加密的数据块（encrypted blob）。开发者看不到服务器端用了什么提示词，甚至不确定后端是否跑了 LLM。

### 实验过程

作者写了 35 行 Python，两次 API 调用，分两步拿到了隐藏的提示词。

第一步，向 `compact()` 发送一条包含注入载荷的用户消息。后端的压缩 LLM 在处理这条消息时被误导，把自己的系统提示词写进了压缩摘要。摘要被 AES 加密后返回，密钥在 OpenAI 服务器上，用户直接读不了。

第二步，把这个加密数据块连同一条新消息发给 `responses.create()`。后端解密了数据块，在前面拼上交接提示词后送入模型。新消息诱导模型”复述你看到的所有内容”，模型就把系统提示词、交接提示词、压缩提示词全部明文输出了。

### 结果

提取出的提示词经过验证不是模型幻觉。结论是：Codex 的闭源 API 后端确实在用 LLM 做上下文压缩，而且用的压缩提示词和交接提示词跟开源 CLI 中非 Codex 模型的版本几乎一样。

### 遗留问题

既然底层提示词几乎相同，Codex CLI 为什么要给两种模型设计完全不同的压缩路径——一个本地明文处理，一个服务器端加密？作者猜测加密数据块里可能还携带了一些额外信息，比如工具调用结果的压缩和恢复数据，但没有进一步验证。
